# 🔄 راهنمای مدیریت Flake، دستور `nh` و جریان کار (`Nix Flake & Workflow Management`)

مدیریت کانفیگ سیستم‌عامل و هوم‌منیجر در پروژه شما بر پایه Flakes و ابزار مدرن **`NH` (Nix Helper)** استوار است. این ابزار سرعت ارزیابی را با استفاده از کش‌های محلی چند برابر کرده و خروجی‌های بسیار خوانا و رنگی (با `nix-output-monitor`) تولید می‌کند.

---

## 🛠️ ۱. بیلد، اعمال و تست سیستم (`NH Commands`)

برای اعمال هرگونه تغییر روی سیستم‌عامل یا هوم‌منیجر، به جای دستورات طولانی سنتی، از میانبرهای شل Fish یا دستورات `nh` استفاده می‌کنیم:

### جدول دستورات بیلد و اعمال کانفیگ

| دستور در شل (`Abbr`) | دستور کامل `nh` | کاربرد و توضیحات |
| :--- | :--- | :--- |
| **`sw`** یا **`nrs`** | `nh os switch` | **بیلد کامل کانفیگ، اعمال فوری روی سیستم زنده و ثبت در منوی بوت‌لودر.** (دستور اصلی روزمره) |
| **`tst`** یا **`nrt`** | `nh os test` | بیلد و اعمال فوری کانفیگ برای تست موقت روی سیستم زنده، **بدون ثبت در منوی بوت‌لودر** (تا اگر ریبوت کردید، کانفیگ قبلی بالا بیاید). |
| **`bld`** یا **`nrb`** | `nh os build` | فقط بیلد کردن کانفیگ و بررسی خطاها بدون اعمال روی سیستم. |
| **`hms`** | `nh home switch` | بیلد و اعمال فقط بخش پروفایل کاربری (`Home Manager`). |

> **💡 نکته خودکارسازی:** در `core.nix` متغیر `NH_FLAKE` به آدرس کانفیگ شما متصل است. بنابراین در هر کجای کامپیوتر که باشید، زدن دستور `sw` کانفیگ مرکزی شما را ارزیابی و بیلد می‌کند.

---

## 🔍 ۲. بازرسی سلامت سینتکس و ارزیابی کانفیگ (`nix-check`)

قبل از اینکه دستور `sw` (`nh os switch`) را بزنید و منتظر بیلد بمانید، می‌توانید در ۳ ثانیه کل سلامت سینتکس، متغیرها و ماژول‌های Flake را ارزیابی کنید!

کافیست در ترمینال دستور زیر را بزنید:
```bash
nix-check
```

### زیر کاپوت `nix-check` چه اتفاقی می‌افتد؟
این اسکریپت ۳ مرحله ارزیابی ایستا و ایزوله را اجرا می‌کند:
1. **`Statix Check`:** کل فایل‌های `.nix` را اسکن کرده و ضدالگوها، سینتکس‌های اشتباه یا پرانتزهای جاافتاده را با شماره خط نشان می‌دهد.
2. **`Deadnix Check`:** متغیرها یا آرگومان‌هایی (`pkgs, lib, ...`) که تعریف کرده‌اید ولی در کد استفاده نکرده‌اید را پیدا می‌کند.
3. **`Flake Evaluation (nix flake check --no-build)`:** منطق ماژول‌های NixOS و Home-Manager را ارزیابی می‌کند و اگر تضادی بین تنظیمات وجود داشته باشد، **قبل از شروع هرگونه دانلود یا بیلد، خطا را اعلام می‌کند.**

## 🔒 ۳. مدیریت ریشه‌های پویا و زباله‌روبی (`Dynamic GC Management`)

همان‌طور که در معماری سیستم دیدیم، سیستم اصلی شما دانلود اولیه برای شل‌ها ندارد (`۰ مگابایت`). اما هر شلی که حتی ۱ بار با دستور `dev <name>` اجرا شود، در مسیر `~/.local/share/dev-roots/` یک لینک پروفایل پویا می‌سازد تا توسط زباله‌روب پاک نشود.

### الف) پاکسازی زباله‌ها و جنریشن‌های قدیمی سیستم
برای پاکسازی پکیج‌های بی‌استفاده و جنریشن‌های قدیمی‌تر از ۱۰ روز (حفظ حداقل ۱۰ جنریشن آخر):
```bash
nh os clean
# یا دستور کلاسیک: sudo nix-collect-garbage -d
```

### ب) پاکسازی و آزادسازی کش شل‌های پویا (`Reclaiming Shell Storage`)
اگر بعد از چند ماه قصد دارید هارد دیسک را کاملاً پاکسازی کنید و شل‌هایی که در گذشته لود کرده بودید (مثلاً `dev rust` یا `dev go`) از حافظه کش پاک شوند:
```bash
# ۱. حذف تمام لینک‌های پروفایل پویا
rm -rf ~/.local/share/dev-roots/*

# ۲. اجرای زباله‌روب برای حذف فوری کامپایلرها از هارد
nh os clean
```

## 🏗️ ۴. شروع پروژه‌های جدید از روی قالب‌های Flake (`Flake Templates`)

در پوشه `templates/` کانفیگ شما، قالب‌های آماده و استاندارد برای شروع پروژه‌های مستقل وجود دارد:

### نحوه شروع یک پروژه جدید (مثلاً پایتون یا Nix)
1. وارد یک پوشه خالی برای پروژه جدید خود شوید:
   ```bash
   mkdir -p ~/projects/my-ai-app && cd ~/projects/my-ai-app
   ```
2. قالب پایتون یا نیکس را از روی کانفیگ خودتان تزریق کنید:
   ```bash
   nix flake init -t ~/projects/nix-config#python
   # یا برای پروژه نیکس: nix flake init -t ~/projects/nix-config#nix
   ```
3. ابزار `direnv` را در پوشه مجاز (`Allow`) کنید:
   ```bash
   direnv allow
   ```
به محض زدن `direnv allow`، محیط مجازی پایتون (`.venv`) همراه با لینترها و LSPهای اختصاصی همان پروژه در پس‌زمینه لود شده و در برابر زباله‌روب محافظت می‌شود!
